# dip 黑膠卡牌遊戲 — 設計藍本

> 這是一份「會慢慢長大」的設計文件，不是規格定稿。實作前先在這裡把規則想清楚，分階段建。
> 現況基礎：前台 SPA（index.html）＋ Cloudflare Worker（dip-vinyl-worker）＋ Firestore（price-manager-e8846）。

---

## 1. 核心概念

> **每一張專輯 = 一張卡。** 玩家透過選歌遊戲「抽到」專輯，收進卡冊，未來可拿卡對戰。

- **卡面**：專輯封面（Spotify）、藝人、專輯名。
- **數值**：冷門度、經典度、入耳難易度（皆 1–5，＋未來可擴充）。
- **稀有度**：由**所有數值加總**決定（總分越高越稀有），**不是單看冷門度**。
- **抽卡來源**：心情選歌 / 類型挑片 / 直接來一張 —— 每次推薦的專輯就是一次「抽卡」。

---

## 2. 資料架構（最重要：未來加數值不用動到玩家舊卡）

### 原則：擁有 與 數值 分離
- **玩家收藏**只記「我擁有哪張卡」：`cardId`、取得時間、張數。**不存任何數值。**
- **卡牌數值**存在一份**共用目錄**（＝ Worker 的 KV 快取，已存在），顯示時即時查。

### 為什麼這樣設計
未來新增第 4、第 5 個數值時 → **只改目錄一處，所有玩家、所有已收藏的舊卡立刻都有**，零遷移。

### 缺值怎麼補：懶人補值（lazy backfill）
數值**不在抽卡當下焊死**，而是在卡片「被顯示時」才向目錄要：
1. 顯示卡片 → 跟 `/album-rating` 要數值。
2. Worker 看 KV：若是舊版、缺新欄位 → **現場算出缺的、寫回 KV**。
3. 回傳完整數值 → 卡片補上。

> 「空值」只存在於「卡更新後第一次被看到」的那一瞬間，補完後全體永久共用。顯示前先 `await` 完整數值即可讓玩家無感（或該格放 shimmer 動畫）。

### cardId 規則
`artist|album` 正規化小寫（與 Worker KV、`/album-rating` 同一把 key），確保目錄與收藏對得起來。

## 3. 玩家身分（不做傳統註冊）

- **起步：Firebase 匿名登入** — 進站自動取得隱形 UID，零註冊就能收藏、存雲端。
- **之後：可綁定 Google/Apple** — 升級匿名帳號做跨裝置同步、可找回。Firebase 全包，不自己寫密碼系統。
- 需在 Firebase Console 啟用：Authentication → 匿名登入。

---

## 4. 卡牌資料模型

### 即時查目錄（會變動、屬於「卡本身」）— 不存玩家收藏
- `coverUrl`、`spotifyUrl`（來自 `/spotify-search`）
- `obscurity` 冷門度（Last.fm listeners）
- `classic` 經典度、`accessibility` 入耳難易度（AI 依樂評共識）
- 未來新增數值都放這層

### 凍結快照（屬於「這次抽卡事件」）— 存玩家收藏
- `cardId`、`obtainedAt`（取得時間）、`count`（重複張數）
- `pulledRarity`（選配）：抽到當下的稀有度，若你希望老卡稀有度不隨聽眾數變動就存它；否則稀有度也走即時查
- `serial`（選配）：流水編號／第幾張抽出

> 判斷原則：**會變且屬於卡本身的 → 走目錄即時查；屬於抽卡事件的 → 存收藏凍結。**

## 5. 稀有度（五階，冷門度降權版）

> **為何不用三項加總？** 遊戲卡池幾乎全是冷門盤，冷門度（obscurity）近乎常數（多半 4–5），拿它當主力會讓所有卡卡在同一格（實測全是獨特）。故改由**經典度＋難度**決定稀有度，冷門度只在「極冷門(=5)」時 +1 加成。

稀有度 **score＝經典度 classic ＋ 難度 accessibility ＋（冷門度=5 則 +1）**（範圍 2–11），越高越稀有。順位高→低：

| score | 稀有度 | 顏色 |
|---|---|---|
| —（無數值） | 殿堂 Hall | 🩶 銀 |
| ≥10 | 傳奇 Legendary | 🟠 橘 |
| 8–9 | 史詩 Epic | 🟣 紫 |
| 6–7 | 獨特 Uncommon | 🟢 綠 |
| 4–5 | 稀有 Rare | 🔵 藍 |
| ≤3 | 普通 Common | ⚪ 灰 |

> **三張「頂點卡」（各把一個數值軸衝到 7 星，互斥、抽中率各 ~3%）：**
> | tier | 名稱 | 衝 7 星的軸 | 判定方式 | 邊框（掃光） | 旗標 |
> |---|---|---|---|---|---|
> | `hall` | 殿堂 | 經典度 | AI 推公認神作（經典模式） | 亮銀 `--silver-shine` | `hall:true` |
> | `pearl` | 流亡 | 冷門度 | 極冷門模式 → **真實聽眾數驗證**（Last.fm 累積 < `ULTRA_OBSCURE_MAX_LISTENERS`，目前 300）才升格 | 暗黑油彩 `--oil-shine` | `pearl:true` |
> | `heresy` | 異端 | 入耳難易度 | AI 推極難入耳的前衛/噪音怪盤（異端模式，主觀軸不驗數據） | 暗赤血光 `--crimson-shine` | `heresy:true` |
> 互斥順序：`classicMode → obscureMode → heresyMode → popularMode`（見 `getGpResult` 內）。設定集中在 `SPECIAL_TIERS`、`ULTRA_OBSCURE_MAX_LISTENERS`（index.html）。註：Spotify「每月聽眾」API 不公開，流亡用 Last.fm 累積聽眾數（worker `/album-rating` 的 `_listeners`）當代理。
> **新手禮：** 剛註冊玩家可免費抽一張隨機殿堂卡，**僅一次**。從 `WELCOME_HALL_ALBUMS` 公認神作名單隨機挑，存成 `hall` 卡，並在 `users/{uid}.welcomeHallClaimed` 標記。卡冊頁（空/非空都會）頂部顯示「🎁 新手禮」橫幅，按鈕觸發 `claimWelcomeHall()`。
>
> **殿堂（銀）= 最稀有、沒有數值。** 不是由 score 決定，而是「經典模式」抽到的公認殿堂神作（OK Computer 等級）才會是殿堂卡。經典模式觸發機率約 3%（`Math.random() < 0.03`），抽到時結果頁顯示銀色「殿堂級」徽章、不顯示星數；收進卡冊時存 `hall: true`，卡冊以銀框＋銀字呈現、無星數。
> **流行模式（普通卡來源）：** 另約 5% 機率（`popularMode`，與經典模式互斥）改推「大眾耳熟能詳的流行常見盤」（Ed Sheeran / Taylor Swift / 周杰倫那種大紅等級）。這類盤冷門度低、難度低、經典度也不高，score 天生低 → 多半落在普通／稀有，用來增加低階卡出現率、平衡稀有度分布。不需特殊標記，走正常 score 計算。
> 另：可在 `RARITY_OVERRIDES`（index.html）手動指定某藝人的稀有度（目前 `dollar brand / abdullah ibrahim → epic`），優先於 score 計算。

> 註：依用戶指定，「獨特綠」順位高於「稀有藍」。
> 例：classic5＋access5＋冷門5 → 11 → 傳奇；classic5＋access3 → 8 → 史詩；classic3＋access2 → 5 → 稀有。
> 程式：`rarityFromRatings()`（index.html），門檻為 config 可調。

### 門檻調校（重要）
`card_catalog` 已記錄每張被抽到的卡，**累積一段時間後可拉真實 score 分布來微調門檻**，確保五色都出現且「越高級越少」。冷門度降權後若仍偏某一格，主要調 classic／accessibility 的門檻即可。

### 未來加數值
若加到第 4、5 個數值（總分上限變 20/25），改用**比例分數** `總和/(數值數×5)` 當門檻，加幾個數值都不必重設。

> 門檻寫成 config，方便調。稀有度由數值即時算出（走目錄），所以加數值後**舊卡稀有度也會自動跟著重算**。

## 8. 對戰系統（PvE，規則 v4 · 品味氣勢制＋屬性效果＋半公開讀心 — 2026-06-26）

> 本節為「專輯卡牌對戰」完整規則。單機網頁、單人對電腦（PvE）、雙方同步回合制。
> 欄位沿用既有目錄：`classic`(經典度)、`obscurity`(冷門度)、`accessibility`(入耳難度，**高＝越難入耳/越前衛**)。
>
> **v3 為何改：** v2 把「宣告屬性」同時當猜拳與傷害數字，導致玩家只挑星星最多的屬性、無視相生相剋。v3 借鏡《三國志X》舌戰/一騎討——**靠讀相剋＋鋪連段充「氣勢槽」，滿了放【品味碾壓】（玩家自選、無視對方屬性的保證一擊）**；星數只剩微薄充能，讀招才是勝負手。已用 `battle.html` Beta 實測，數值為 playtest 值、可調。

### 8.1 基礎參數
- **勝利條件**：雙方起始「品味聲望（HP）」**30 點**，先把對手扣到 ≤0 者勝。
- **卡冊 / 牌組 / 牌庫**（三層分離）：
  - **卡冊**＝玩家全部收藏（＝既有 `collections/{uid}/cards`），是資產、不直接上場。
  - **牌組**＝開戰前從卡冊挑 **20 張**組成。
  - **牌庫**＝牌組洗牌後的抽牌堆，對局抽牌一律從牌庫抽。
- **牌組限制**：固定 20 張；同名常規卡 ≤2 張；王牌（單軸 7）整副 ≤1 張；只能放卡冊實際擁有的卡。
- **手牌**：起手 5 張；每回合開始抽 1；回合結束手牌 >7 須棄到 7。

### 8.2 卡牌數值
- 常規卡三軸 `classic` / `obscurity` / `accessibility`，單軸上限 **5**。
- **王牌（單軸 7）＝既有頂點卡旗標**，對戰用三圍如下（卡冊顯示端可仍不秀數值，數值只在對戰生效）：

| 旗標 | 卡名 | 對戰三圍（classic/obscurity/accessibility） |
|---|---|---|
| `hall` | 殿堂 | **7 / 1 / 1** |
| `pearl` | 流亡 | **1 / 7 / 1** |
| `heresy` | 異端 | **1 / 1 / 7** |

### 8.3 雙資源：品味聲望（HP）＋ 氣勢槽
- 每邊除了 **HP 30**，另有 **氣勢槽 0→6**（起始 0）。
- **傷害＝星數驅動，讀招決定打不打中、星數決定多痛**：相剋命中扣血 = 勝方那招星數（1–7）；純比星扣血 = 星數差；品味碾壓固定 −6（無視屬性）。→ 讀招（命中）與星數（傷害）兩個維度都要算，既不是「只比星數大小」、也不是「星數無用」。
- **氣勢只進不退**（不被相剋倒扣，僅王牌可清/壓）：相剋勝 +3、被剋方仍 +1 安慰、數值勝 +1、平手雙方 +1、連段 +1/+2（第 4 環起歸零）；故槽永遠在動、可逆轉。

### 8.4 雙循環文化邏輯（核心）
每回合雙方各打 1 張卡，並宣告一項「主打屬性」。雙方同時亮牌後依序結算。

> **三軸的文化定位**（相生相剋的邏輯都建立在此）：
> - **經典 classic**＝大眾公認、歷史留名的主流名盤（靠時間驗證的地位）。
> - **冷門 obscurity**＝地下小眾、稀有私盤（靠「沒人聽過」的資訊差與神祕感）。
> - **入耳 accessibility**＝其實是「入耳<u>難度</u>」＝前衛實驗的硬度，**越前衛／越解構＝星數越高**（非「好聽／洗腦」，勿搞反）。

**相生（自我充能，循環：經典 → 冷門 → 入耳 → 經典）**——宣告屬性接續上一回合者觸發；這條鏈是一個聽眾的**進化史**：
- **【品味拓荒】** classic → obscurity：大眾名盤聽到審美疲勞，轉身往地下挖未知私盤。
- **【閾值突破】** obscurity → accessibility：連冷門都餵不飽耳朵，刺激門檻越拉越高，走向極端實驗聲響。
- **【前衛收編】** accessibility → classic：玩過最極端的解構後返璞歸真，用全新眼光重新確立主流經典的偉大。
- 充能：2 連鎖 **氣勢 +1**；3 連鎖 **氣勢 +2**；**第 4 環起不再充能**（須斷掉重鋪，避免無腦循環刷氣勢）；中斷或屬性重複 → 連鎖歸零。

**相剋（對手壓制，循環：冷門 剋 經典 → 經典 剋 入耳 → 入耳 剋 冷門）**——每組剋制都有文化上的道理：
- **【品味突襲】** obscurity 剋 classic：拿邊緣稀有的冷門盤，戳主流名盤「太通俗」的痛點——你聽的大家都聽過，有什麼了不起？
- **【大眾共鳴】** classic 剋 accessibility：經得起時間的歷史地位，正面壓過前衛的晦澀與抗拒感——再前衛也撼動不了公認的偉大。
- **【降維打擊】** accessibility 剋 obscurity：用前衛的硬實力與混亂，拆穿小眾「只是沒人聽過」的神祕防線——稀有不等於厲害。

**屬性專屬效果（v4 新增，給「非最高星」屬性一個選它的理由）**——宣告該屬性即觸發（固定值、不隨星數放大，故選屬性看戰術而非比星數）：
- **經典〔守〕**：自己 **HP +1**（厚度／聲望）
- **冷門〔挖〕**：自己 **氣勢 +1**（資訊差）
- **入耳〔攻〕**：對手 **氣勢 −1**（前衛壓制）

> **相生與相剋方向相反**，照相生鋪連段不會正好餵給對手相剋方向，避免「順著走＝被預判」的死亡螺旋。
> **星數＝傷害口徑（非選屬性的「比大小」依據）**：選哪個屬性看戰術（讀相剋／續連段／屬性效果），但**一旦命中，扣多少血由那招星數決定**——所以你會想「用高星的屬性去剋」，可是無腦挑高星仍會被讀招虛晃打爆。

### 8.5 結算（氣勢 + 傷害）
比對雙方宣告屬性：
- **相剋時**：剋制方 **氣勢 +3**、被剋方 **氣勢 +1**（安慰、不倒扣），且被剋方 **HP − 剋制方那招的星數**（讀招命中，傷害＝口徑；普卡 1–5、王牌單軸 7）。
- **無相剋（同屬性或無關）**：純比面板星數，高的一方 **氣勢 +1**、且 **對手 HP −（星數差 × valueDmg，預設 ×1）**；平手雙方 **氣勢 +1**、不扣血。（`valueDmg` 設 0 即回到「純比星只給氣勢、不扣血」。）
- **品味碾壓**：**氣勢槽滿（≥6）時，玩家可自選**對一張卡發動——**無視對方屬性的保證一擊，對手 HP −6**，自身氣勢清零（電腦則在氣勢滿時自動發動）。← 主導權在玩家、週期性爆發。
- 雙殺（雙方同回合 ≤0）→ 剩血多者勝，無平手。

### 8.6 王牌被動（重綁氣勢經濟）
- **共通【霸權】**（hall/pearl/heresy）：作為被剋方時，**取消該次相剋**（對手不得 +3 氣勢與 −3 壓制，落入純面板比氣勢）。王牌＝難以充能的硬壁壘。
- **殿堂【眾望所歸】**：打出回合，**對手氣勢 −2**（主流壓制，延遲對手碾壓）。
- **流亡【私盤特攻】**：作為剋制方時，相剋充能 +3 → **+4**。
- **異端【終極解構】**：打出時立即**歸零對手整條氣勢槽**（解構核爆；於相剋結算前先生效）。

### 8.7 半公開讀心（v4 PvE：氣場提示 ＋ 虛晃反咬）
> 為了讓玩家「讀招」而非「比星數」，PvE 改成**電腦先暗中決定出招、並對玩家露出一個「氣場提示」**（指向它的傾向屬性，如「露出考古挖盤的眼神」＝偏冷門）。**仍是暗選**（不亮牌），玩家依提示判斷要不要去剋。
> **聰明點在「虛晃」**：提示誠實，但**玩家上一手若照提示去剋，電腦這手就虛晃**——改打「能剋掉玩家預期 counter」的屬性反咬。故無腦照提示剋會被狂 punish（模擬：無腦剋只贏 1%、會讀的贏 ~67%），逼玩家真的讀。
> 工程：`startTurn()` 先 `aiPlan()` 出 `pendingE{card,attr,tell,ult}`；`pCounteredTellLast` 為真則把 `attr` 改成 `COUNTER_OF[COUNTER_OF[tell]]`。

### 8.8 回合流程
1. **電腦暗定＋露提示**：電腦先決定出招，露出氣場提示（會依玩家上手是否照剋而虛晃）。
2. **抽牌**：各抽 1；牌庫已空 → 觸發疲勞（見 8.9）。
3. **選曲**：玩家讀提示，暗選 1 卡 ＋ 主打屬性（或氣勢滿時選品味碾壓）。
4. **亮牌**：公開（王牌即時被動：異端歸零氣勢、殿堂壓氣勢先生效）。
5. **相生 → 相剋／數值 → 屬性效果**：依序算氣勢增減與壓制傷害。
6. **品味碾壓**：氣勢滿者發動（玩家自選、電腦自動），無視對方屬性一擊。
7. **棄牌**：手牌 >7 棄到 7。回到 1。

### 8.9 牌庫耗盡（疲勞傷害）
牌庫空、無牌可抽時：該方該回合不補牌並受疲勞傷害，**首次 1 點、之後每次遞增 1**（1/2/3…），不可被減免，可達成 HP≤0 判負。對局拉長後加速收尾。（電腦手牌也出完時 → 顯示「疲勞無牌」，該回合只結算玩家側。）

### 8.10 工程備註
- **先做 PvE**：幾乎純前端、零後端，先驗證好不好玩；PvP（配對/同步/防作弊）確認有趣後再評估。已有可玩 Beta：`battle.html`。
- **AI 難度**建議分級：簡單＝隨機合法出牌；普通＝追連段＋讀心虛晃（Beta 現況）；困難＝更精準的虛晃時機與資源管理。
- 對戰參數全部寫成 config（以下為 Beta playtest 值，可調）：

```js

  startHp: 30, deckSize: 20, startHand: 5, drawPerTurn: 1, handLimit: 7,
  maxCopies: 2, aceCopies: 1, regularStatCap: 5, aceStatValue: 7,

  beats: { obscurity: 'classic', classic: 'accessibility', accessibility: 'obscurity' },
  // 相生：接續 chainNext 即連段（與 beats 反向）
  chainNext: { classic: 'obscurity', obscurity: 'accessibility', accessibility: 'classic' },

  // 氣勢經濟（只進不退，僅王牌可清/壓）

  qiValueLoser: 0,    // 數值敗充能

  // 傷害＝星數驅動：相剋命中扣血 = 勝方那招星數(1–7)；純比星扣血 = 星數差 × valueDmg
  valueDmg: 1,        // 純比星傷害倍率（設 0 = 純比星只給氣勢、不扣血）
  ultimate: 6,        // 品味碾壓傷害（無視對方屬性、保證命中）
  attrFx: { classic: 1, obscurity: 1, accessibility: 1 }, // 屬性效果：經典守+HP / 冷門挖+氣勢 / 入耳攻-對手氣勢

  // 半公開讀心：電腦露出 tell；玩家上手照剋 → 本手虛晃打 COUNTER_OF[COUNTER_OF[tell]] 反咬
  // （見 8.7。COUNTER_OF[x] = 能剋掉 x 的屬性，由 beats 反推）

  // 王牌：三圍 + 氣勢向被動
  ace: {
    hall:   { stats: { classic: 7, obscurity: 1, accessibility: 1 }, enemyQi: 2 },   // 眾望所歸：對手氣勢 -2
    pearl:  { stats: { classic: 1, obscurity: 7, accessibility: 1 }, counterQiBonus: 1 }, // 私盤特攻：相剋多 +1 氣勢
    heresy: { stats: { classic: 1, obscurity: 1, accessibility: 7 }, zeroEnemyQi: true }, // 終極解構：歸零對手氣勢
  },
  hegemonyCancelsCounter: true, // 霸權：王牌被剋時取消該次相剋

  fatigueStart: 1, fatigueStep: 1,
};
```

collections/{uid}/cards/{cardId}

### 安全規則（草案，需在 Console 發布）
```
match /collections/{uid}/cards/{cardId} {
  allow read, write: if request.auth != null && request.auth.uid == uid;
}
match /users/{uid} {
  allow read, write: if request.auth != null && request.auth.uid == uid;
}
```
> 卡牌數值不放 Firestore（走 Worker KV 目錄），所以規則只要管「玩家自己的收藏」。
